## 启动/停止脚本自愈加固（兜底逻辑写死，部署机最小卡顿）
- 需求：把全部兜底/自愈逻辑写进固定脚本，部署机操作最小卡顿；解决 `OSError: [Errno 48] Address already in use`（端口被残留进程占）。
- 根因复盘：Errno 48 多因「直接 `python -m app.amazon_toolbox.server` 起的孤儿实例」未被 pid 文件管理，重启时端口冲突。旧 start.sh 依赖 lsof/fuser（macOS/Linux 行为不一致）且 `set -u` 下 `PORT` 变量处理有 bug。
- 修复（4 个脚本全部重写并实测通过）：
  - `start.sh`/`stop.sh`：用 `pgrep -f "app.amazon_toolbox.server"` / `pkill -f` 按进程名杀**所有**实例（覆盖 start.sh / 直接 python -m / 重复启动）；pid 文件 + 端口探测兜底。
  - **安全杀**：杀 pid 前先 `pgrep -f` 校验该 PID 确属本服务进程，避免 PID 被复用后误杀无关进程。
  - **bash 3.2 nounset 坑**：macOS 默认 `/bin/bash` 是 3.2.57，`set -u` 下 `$PORT` 在部分字符串上下文会误报 unbound（同文件 line 105 正常、line 91 报错，典型 3.2 行为）。所有字符串/echo 中的 `$PORT` 改为 `${PORT:-8080}`（PORT 恒由 line 16 兜底，安全）。
  - `start.sh`：端口就绪用 bash `/dev/tcp` 探测（不依赖 curl）；端口仍占用时仅告警不阻塞。
  - `start.ps1`/`stop.ps1`：pid 文件停 + `Get-NetTCPConnection`/`Stop-Process` 端口释放 + `Get-CimInstance` 命令行校验（仅当命令行含 `app.amazon_toolbox.server` 才杀），与 bash 行为对齐。
- 实测（bash 3.2，set -euo pipefail）：① 模拟孤儿实例→`start.sh` 自动 `停止匹配进程` 并干净重启，/api/health={ok:true}，无 Errno 48；② 连跑两次 `start.sh` 幂等重启正常；③ `stop.sh` 干净停进程、释放端口、删 pid 文件。`bash -n` 全部通过。

## 汇总报告「店铺」列值错误修复（_infer_store_from_path）
- 现象：结果 Excel 中「店铺」列显示 `"2026年1-3月及6月（按月下载）"`（最外层日期文件夹名），而非品牌名（如 Anker）。
- 根因：`_infer_store_from_path()` 从目录**第 1 层（最外层）向内遍历**，第一个非国家文件夹即被返回为品牌名——日期文件夹排在品牌前面，所以被错误选中。
- 修复：改为**从最内层向外遍历（reversed）**，并增加日期/期间类文件夹过滤模式（匹配 `年/Q/下载/汇总/报告` 等）。优先级变为：品牌(最靠近 PDF) > 其他非国家非日期文件夹 > 最外层兜底。
- 测试覆盖 8 个场景全部通过，含截图中的实际结构 `["2026年1-3月...", "美国", "Anker"] → 正确返回 `Anker`。

## 修复：品牌名应从 PDF 文件名提取而非目录路径（Bikoney 案例）
- 现象：文件 `202606-Bikoney-加拿大-汇总报告.pdf` 的「店铺」列未显示 Bikoney，而是显示了目录路径中的某个文件夹名。
- 根因：`collect_pdfs()` 中 `qname` 正则仅匹配旧格式 `STORE-XX-YYYYQn`；用户实际文件名为 `YYYYMM-BRAND-COUNTRY-TYPE.pdf` 格式，qname 不匹配 → 静默回退到 `_infer_store_from_path()` 从目录路径推断品牌——取到了错误的值。

  1. qname（旧格式 STORE-XX-YYYYQn）→ 2. fn_brand（新格式 YYYYMM-BRAND）→ 3. _infer_store_from_path（目录路径兜底）
